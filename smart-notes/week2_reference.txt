--------------------------------------------------------------------------------
  WHAT WE BUILT
--------------------------------------------------------------------------------

A persistent, semantically searchable CLI notebook.
Notes survive program restarts. Search by meaning, not keywords.

Commands:
  python3 main.py add "note text" --tag work
  python3 main.py search "your query" --top 3
  python3 main.py list
  python3 main.py delete note_20260413_183748_144129

--------------------------------------------------------------------------------
  THE PROBLEM WEEK 1 HAD
--------------------------------------------------------------------------------

Week 1 stored embeddings in a Python list (RAM).
Restart the program → list is gone → re-embed everything from scratch.
10 snippets = fine. 10,000 documents = hours of wasted work.

Week 2 fix: ChromaDB persists embeddings to disk in chroma_db/.
Embed once. Load instantly on every future run.

--------------------------------------------------------------------------------
  KEY CONCEPT 1: VECTOR DATABASE
--------------------------------------------------------------------------------

A regular database answers: "find rows WHERE tag = 'work'"  → exact match
A vector database answers: "find rows NEAREST to this query" → semantic match

When you add a document, it stores three things together:
  1. The original text     → so you can display it
  2. The embedding         → so it can do similarity math
  3. Metadata (dict)       → tags, timestamps, any extra info

When you query, it embeds your query and finds the nearest stored embeddings.
The manual loop from Week 1's similarity.py? The DB does it internally.

--------------------------------------------------------------------------------
  KEY CONCEPT 2: CHROMADB
--------------------------------------------------------------------------------

Simplest local vector database. No server, no Docker, no config.
Just a Python library that saves everything to a folder on disk.

Two client modes:
  chromadb.Client()                → in-memory only (dies on restart, like Week 1)
  chromadb.PersistentClient(path)  → saves to disk (what we use)

Collections = tables for vectors. Created once, reloaded automatically.
  client.get_or_create_collection(
      name="notes",
      metadata={"hnsw:space": "cosine"}   ← IMPORTANT: set distance metric
  )

Three core operations (same API in every vector DB, including pgvector):
  collection.add()     → store text + embedding + metadata
  collection.query()   → semantic search
  collection.delete()  → remove by ID
  collection.get()     → fetch all (no embedding needed)
  collection.count()   → how many items stored

--------------------------------------------------------------------------------
  KEY CONCEPT 3: DISTANCE vs SIMILARITY
--------------------------------------------------------------------------------

ChromaDB returns DISTANCE, not similarity. Opposite scale:

  Similarity: 1.0 = identical,  0.0 = unrelated  (goes DOWN as things differ)
  Distance:   0.0 = identical,  2.0 = opposite    (goes UP as things differ)

Why distance? Databases are built around "nearest neighbor" search.
Smallest distance = most similar. Same as GPS finding the nearest restaurant.

Conversion formula (cosine distance → cosine similarity):
  cosine_distance = 1 - cosine_similarity
  → cosine_similarity = 1 - cosine_distance

ChromaDB's cosine distance range is 0-2 (not 0-1), so we normalize:
  similarity = 1 - (dist / 2)

  distance 0.0 → similarity 1.00  (identical)
  distance 0.5 → similarity 0.75  (very similar)
  distance 1.0 → similarity 0.50  (unrelated)
  distance 2.0 → similarity 0.00  (opposite)

--------------------------------------------------------------------------------
  KEY CONCEPT 4: DISTANCE METRIC — WHY "hnsw:space": "cosine" MATTERS
--------------------------------------------------------------------------------

ChromaDB default metric: L2 (Euclidean distance)
  → Measures straight-line distance between two points in 768D space
  → Values can be huge (500+)
  → Our formula breaks: 1 - (500/2) = nonsense

Cosine distance:
  → Measures the ANGLE between two vectors (direction, not magnitude)
  → Range always 0 to 2
  → Same concept as Week 1's cosine similarity
  → Correct for text embeddings — we care about meaning, not vector length

IMPORTANT: Set the metric at collection creation time.
You cannot change it on an existing collection.
If you change it, delete chroma_db/ and start fresh.

--------------------------------------------------------------------------------
  KEY CONCEPT 5: ARGPARSE SUB-COMMANDS
--------------------------------------------------------------------------------

argparse sub-commands work like git: "git add", "git commit", "git log"

  subparsers = parser.add_subparsers(dest="command", required=True)
  p_add = subparsers.add_parser("add")

Two argument types:
  Positional (no --):  p_add.add_argument("text")         → required, typed directly
  Optional flag (--):  p_add.add_argument("--tag")        → optional, uses --name value

  python3 main.py add "note text"           ← "note text" is positional
  python3 main.py add "note text" --tag work ← --tag is an optional flag

set_defaults(func=cmd_add) links sub-command to its handler.
args.func(args) calls the right handler — no if/elif chain needed.

--------------------------------------------------------------------------------
  BUGS CAUGHT AND FIXED
--------------------------------------------------------------------------------

1. results["id"] vs results["ids"]
   ChromaDB always uses plural "ids". Singular raises a KeyError.

2. Missing "similarity" key in search results
   The loop built dicts but forgot to include the similarity score.
   main.py crashed trying to display it.

3. include=["id"] in collection.get()
   ChromaDB's .get() doesn't accept "id" in include list.
   IDs are always returned automatically — just omit it.

4. Wrong distance metric (L2 default)
   Default metric gives huge distances, breaking the similarity formula.
   Fix: metadata={"hnsw:space": "cosine"} at collection creation.
   Then delete chroma_db/ to recreate with the new metric.

5. args.top_k vs args.top
   argparse stores --top as args.top, not args.top_k.
   Attribute name = flag name, not your variable name.

  ├── requirements.txt   ← chromadb, ollama, and their dependencies

--------------------------------------------------------------------------------
  SEMANTIC SEARCH BEHAVIOR — IMPORTANT NOTES
--------------------------------------------------------------------------------

1. There is no "no match". Every query always returns results.
   Embeddings always have an angle between them. The DB always finds something.
   In production, set a threshold (e.g. similarity < 0.75 = "no relevant results").

2. Scores only mean something relative to each other.
   With 1 note, every query returns that note with the same score.
   With 10+ notes, high vs low scores become meaningful.

3. Embeddings encode meaning, not keywords.
   "Porject" (typo) finds "Work on side project" because the model understands
   the intent, not the characters. This is the power over keyword search.

4. Scores are compressed in small datasets.
   0.87 vs 0.82 with 2 notes = almost meaningless difference.
   0.95 vs 0.61 with 20 notes = clear signal.
